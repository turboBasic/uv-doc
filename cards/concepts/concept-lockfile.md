## Summary

`uv.lock` is uv's universal, cross-platform lockfile: it pins the exact set of packages
that would be installed across all supported platforms and Python versions, giving
reproducible environments. It is generated by `uv lock` and consumed by `uv sync`.

## Syntax / Usage

```bash
uv lock              # resolve dependencies and write/update uv.lock
uv sync              # make the environment match uv.lock
```

## Details

`uv.lock` is human-readable TOML but **must not be hand-edited** — uv manages it. It is
"universal" in that a single lockfile captures resolutions for every Python marker, not
just the current machine. Check it into version control for reproducibility.

`uv lock` and `uv sync` are complementary: `lock` resolves and records versions; `sync`
installs exactly what the lockfile records into `.venv`. Both run automatically when you
invoke `uv run`.

Lockfile freshness flags:

- `--locked` — error if `uv.lock` is outdated rather than updating it (CI guard).
- `--frozen` — use `uv.lock` without checking whether it is up-to-date.

uv treats the lockfile as outdated when dependencies are added/removed in
`pyproject.toml` or when constraints change so the locked versions no longer satisfy
them. Critically, uv does **not** consider the lockfile outdated merely because newer
package releases exist — upgrades require explicit action.

## Examples

```bash
# Upgrade everything to the newest allowed versions
uv lock --upgrade

# Upgrade a single package
uv lock --upgrade-package requests

# Pin a package to a specific version during upgrade
uv lock --upgrade-package requests==2.31.0

# CI: fail if the committed lockfile is stale
uv sync --locked
```

## Caveats / Common Mistakes

- Releasing a new upstream version does not auto-update the lock; run `uv lock --upgrade`
  or `--upgrade-package <name>` deliberately.
- Do not edit `uv.lock` by hand — change `pyproject.toml` and re-lock instead.
