## Summary

`uv_build` is uv's native PEP 517 build backend for pure-Python projects. It is the default
backend generated by `uv init --package` and integrates tightly with uv for fast, validated builds
with zero configuration for most projects.

## Syntax / Usage

```toml
# pyproject.toml
[build-system]
requires = ["uv_build>=0.11.24,<0.12"]
build-backend = "uv_build"
```

## Details

### What uv_build is

`uv_build` is a PEP 517-compliant build backend distributed as a standalone PyPI package. It
transforms a source tree into a source distribution (`sdist`) and a wheel. uv supports any PEP 517
build backend, but `uv_build` is the native choice: it validates project metadata and structure,
produces helpful error messages, and is very fast.

### Enabling it in `[build-system]`

Add the `[build-system]` table to `pyproject.toml` with `uv_build` as both the required package
and the backend entry point. `uv init` (with `--package` or `--build-backend uv_build`) writes this
table automatically.

The presence of a `[build-system]` table tells uv that the project is a package that should be
built and installed into the project virtual environment. Without it, uv treats the project as
virtual — only its dependencies are installed.

### Versioning and the upper-bound requirement

`uv_build` follows the same versioning policy as uv itself: the minor version is bumped for
breaking changes; patch versions are for bug fixes and non-breaking enhancements. Because minor
bumps may introduce breaking changes, the `requires` constraint should include both a lower bound
and an upper bound on the minor version, e.g. `>=0.11.24,<0.12`. This ensures the package
continues to build correctly as new `uv_build` versions are released.

### Bundled vs. package

The `uv` executable ships with a bundled copy of `uv_build`. When `uv build` runs and the bundled
version satisfies the `uv_build` requirement declared in `[build-system]`, uv uses the bundled copy
directly — no separate download needed. If the bundled version is outside the declared range, uv
downloads and uses a compatible `uv_build` package from PyPI instead.

Other build frontends (e.g., `pip`, `python -m build`) always use the `uv_build` package from
PyPI, selecting the latest compatible version.

### Pure-Python-only constraint

`uv_build` only supports pure-Python packages. Projects with C/Cython extension modules or other
native code must use a different build backend (e.g., `hatchling`, `setuptools`, `meson-python`).
Similarly, projects that require a build script (`build.py`) for flexible project layouts should
use an alternative backend such as `hatchling`.

## Examples

Enable uv_build in an existing project:

```toml
# pyproject.toml
[build-system]
requires = ["uv_build>=0.11.24,<0.12"]
build-backend = "uv_build"
```

Create a new packaged project using uv_build:

```bash
uv init --package my-lib
```

Create a new project explicitly requesting the uv build backend:

```bash
uv init --build-backend uv_build my-lib
```

Build the project (sdist + wheel into `dist/`):

```bash
uv build
```

## Caveats / Common Mistakes

- **Always include an upper bound** on `uv_build` in `requires`. Without `<0.12` (or the equivalent
  next minor), a future minor release may introduce breaking changes that silently break your build.
- **Extension modules are not supported.** Attempting to use `uv_build` for a package with compiled
  code will fail. Use `setuptools`, `meson-python`, or another backend instead.
- **No build scripts.** `uv_build` does not support `build.py`-style customization. For complex
  project layouts that need a build script, use `hatchling`.
- **Other frontends ignore the bundled backend.** When building via `pip` or `python -m build`,
  the bundled uv backend is not used — only the `uv_build` PyPI package is considered.
